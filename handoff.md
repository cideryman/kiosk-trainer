# 배달왔삼 키오스크 프로젝트 핸드오프

> 마지막 갱신: 2026-07-21

## 현재 상태

- **치명 이슈**: 현재 재현 가능한 치명 버그나 미해결 보안 실패는 없습니다.
- **운영 DB 기준**: `주간보호 매점DB (2).xlsx`의 실제 시트 10개와 1행 헤더 대조 및 신규 `이용신청` 22열(A:V) 시트의 생성 및 운영 검증을 완료했습니다.
- **GAS 구조**: `gas/*.gs` 기능별 분리를 완료했으며 같은 Apps Script 프로젝트 안에서 함께 배포합니다.
- **정적 캐시**: `service-worker.js`의 현재 캐시 버전은 `kiosk-cache-v226`입니다.
- **최근 구현**: 배달왔삼 이용 신청 공개 페이지, 관리자 신청 탭, `이용신청` 22열 계약, 멱등 접수, 중복 연락처 차단, 30일 만료 익명화, 기본 5명·1~100명 정원 조절, 대기자 기능 및 발달장애인 AAC 답글 UX, 내 주문 내역 후기 상태 보존(P22)을 완료했고, 직원 후기 관리 화면의 AAC 프리셋 복구 및 TTS 이모티콘 필터링 정제(P27)를 완료했으며, 답글 작성 2x2 이모티콘 그리드 개편 및 화면별 이미지 렌더링 연동(P28)을 완료하고, 관리자 페이지(admin.html) CSS/JS 외부 파일 분리 리팩토링(P29), 주방 페이지(kitchen.html) CSS/JS 외부 파일 분리 리팩토링(P30), 게스트 주문 화면(guest.html) CSS/JS 외부 파일 분리 리팩토링(P32), 직원 후기 관리 화면(reviews.html) CSS/JS 외부 파일 분리 리팩토링(P33), 지난 주문 보관 명시적 확인 오류 조치(P31), 내 주문 확인 화면 지난 보관 주문 더보기(Lazy Loading - P40), 장애인식 개선 캠페인/행사용 간식 모드 스위칭(P24) 및 관리자 게스트/행사 간식 및 순서 3단 분리 개편(P41)을 완료했습니다.
- **외부 상태 주의**: 저장소에서는 GAS 실제 배포 여부와 운영 시트 생성 여부를 확인할 수 없습니다. 사용자 확인 전에는 항상 `배포·운영 검증 대기`로 기록합니다.

## AI 작업 원칙

1. **현재 상태와 다음 작업은 이 파일만 기준으로 판단합니다.**
2. 작업 영역에 따라 `architecture.md`, `decisions.md`, `database-schema.md`, `verification.md`만 추가로 읽습니다.
3. 실제 DB 구조를 추측하지 않습니다. 구조 변경 전에는 최신 DB 사본 또는 운영 점검 결과를 확인합니다.
4. 운영 중인 시트 열을 코드 가정에 맞춰 임의로 이동하거나 삭제하지 않습니다.
5. GAS를 수정하면 결합 구문 검사 후 새 버전을 배포하고, 정적 파일을 수정하면 서비스 워커 캐시 버전 상향 필요성을 확인합니다.
6. 모든 화면 변경은 [모바일 친화적 개발 원칙](docs/handoff/decisions.md#모바일-친화적-개발-원칙)을 확인하고 PC·390px·320px 회귀 범위를 함께 정합니다.
7. 완료 작업의 결과는 `work-log.md`에 한 번만 남기고, 긴 분석 원문은 `handoff-archive.md`로 보냅니다.
8. 문서 변경을 마치기 전에 `node scripts/check-handoff.js`를 실행합니다.

## 핸드오프 기록 안전 규칙

### 1. 작업 번호

- 작업 번호는 한 번 사용하면 완료·보류·폐기 후에도 다른 작업에 재사용하지 않습니다.
- 현재 사용이 확정된 마지막 번호는 `P41`이며, **새 작업은 `P42`부터 순서대로 부여합니다.**
- 기존 작업의 내용이 달라지면 같은 번호를 재활용하지 않고 새 번호를 발급한 뒤 이전 작업과의 관계를 적습니다.

### 2. 허용 상태

작업 상태는 다음 표현만 사용합니다.

`검토 대기` → `구현 대기` → `구현 중` → `구현 완료·배포 대기` → `배포 완료·수동 검증 대기` → `완료`

진행하지 않기로 한 작업은 `보류` 또는 `폐기`로만 기록합니다.

- 자동 테스트 통과를 `수동 검증 완료`로 기록하지 않습니다.
- 코드가 저장소에 있어도 GAS 실제 배포를 확인하지 못했다면 `배포 대기`를 유지합니다.
- 사용자가 배포 또는 수동검증 완료를 명확히 확인했을 때만 다음 상태로 바꿉니다.

### 3. 파일별 역할

- `handoff.md`: 현재 상태와 작업 상태를 기록하는 유일한 기준입니다. 완료·폐기 작업은 현재 표에 남기지 않습니다.
- `work-log.md`: `완료` 처리된 작업의 날짜와 결과만 기록합니다. 대기·보류 상태는 적지 않습니다.
- `verification.md`: 상태 표현 없이 반복해서 사용할 검증 절차만 기록합니다.
- `decisions.md`: 장기간 유지할 정책과 폐기한 대안의 이유를 기록합니다.
- `database-schema.md`: 실제 확인한 DB 구조와 아직 배포 전인 코드 계약을 명확히 구분합니다.
- `active-work.md`는 더 이상 사용하지 않습니다. 이전 내용은 날짜가 붙은 아카이브에서만 확인합니다.

### 4. 상태 변경 시 동시 갱신

- **새 작업·우선순위·배포·수동검증 상태 변경**: `handoff.md`의 현재 우선순위 표만 갱신
- **작업 완료**: 활성 목록에서 제거하고 `work-log.md`에 결과 추가
- **정책 변경**: `decisions.md`
- **DB 계약 변경**: 최신 헤더를 확인한 뒤 `database-schema.md`
- **에셋 추가·교체**: `asset-sources.md`

### 5. 기록 전 최종 점검

1. 같은 작업 번호가 다른 의미로 사용되지 않았는지 확인합니다.
2. 현재 우선순위 표에 `완료`·`폐기` 항목이 남아 있지 않은지 확인합니다.
3. `verification.md`에 `완료`·`대기` 상태를 기록하지 않았는지 확인합니다.
4. 실제 배포·수동검증을 추측해 완료 처리하지 않았는지 확인합니다.
5. 서비스 워커 캐시 버전과 DB 열 수처럼 코드로 확인 가능한 값은 실제 파일과 대조합니다.
6. `node scripts/check-handoff.js`가 성공하는지 확인합니다.
