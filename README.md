# 삼각지 카페 키오스크 & 배달왔삼

주간보호센터의 **발달장애인 키오스크 훈련**과 지역사회 음료 배달 서비스 **배달왔삼**을 함께 운영하는 정적 웹앱입니다. 프론트엔드는 GitHub Pages와 같은 공개 정적 호스팅에서 제공하고, 주문·재고·크레딧·관리 기능은 Google Apps Script(GAS)와 Google Sheets를 통해 처리합니다.

## 프로젝트 목적

### 일반 키오스크 훈련

- 이용자가 간식 선택, 크레딧 사용, 주문 확인과 완료 과정을 반복 연습합니다.
- 실제 기관 태블릿에서 사용할 수 있도록 큰 터치 영역, 사진·이모지, 음성 안내와 햅틱 피드백을 사용합니다.
- 공개 GitHub 주소에서 운영되므로 이용자 식별 정보는 현재 `닉네임` 중심으로 표시합니다.

### 배달왔삼

- 봉사자·후원자 등이 사전 이용 신청 후, 운영일에 음료를 주문하고 배달받는 서비스입니다.
- 발달장애인 이용자가 주문 확인, 음료 준비와 전달 과정에 참여합니다.
- 게스트·카카오 로그인, 일일 크레딧, 포장·배달, 주문 조회·취소, 후기와 이용 신청·승인 관리가 포함됩니다.

두 모드는 주문·재고·관리 화면을 일부 공유하지만, **일반 키오스크 이용자 데이터와 배달왔삼 게스트 데이터는 별도 흐름**으로 처리합니다.

## 주요 화면

| 화면 | 역할 |
| --- | --- |
| `index.html` | 일반 키오스크 이용자 선택 및 훈련 시작 |
| `menu.html` | 일반 키오스크 또는 `?browse=guest` 배달왔삼 간식 선택 |
| `confirm.html`, `complete.html` | 주문 확인·수령 방식 선택·주문 완료 |
| `guest.html` | 배달왔삼 운영 상태, 카카오 연결, 주문 시작과 이용 안내 |
| `guest-orders.html` | 배달왔삼 주문 조회·취소 |
| `guest-apply.html` | 배달왔삼 사전 이용 신청 |
| `admin.html` | 이용자·간식·순서·이용 신청 관리 |
| `kitchen.html` | 주문 처리, 제공 완료, 통계와 재고 확인 |
| `reviews.html` | 후기 검수·공개 상태·답글 관리 |
| `board.html` | 준비 중·준비 완료 주문을 보여주는 전광판 |
| `print-bills.html` | 빌지와 배송 체크리스트 인쇄 |

관리자·주방·후기 화면은 관리자 인증 게이트를 거친 뒤 데이터를 표시합니다. 일반 키오스크와 공개 배달왔삼 페이지는 운영 목적상 공개 진입 흐름을 유지하며, 주문·재고·크레딧 차감은 서버에서 최종 확인합니다.

## 핵심 기능

- **일반 키오스크**: 이용자·간식 선택, 크레딧 차감, 주문 확인, 완료 화면
- **배달왔삼 주문**: 포장·배달 선택, 배달지 입력, 게스트·카카오 사용자 식별, 주문 토큰과 주문 조회
- **멱등 주문**: `idempotencyKey`로 네트워크 재시도와 중복 클릭에 따른 동일 주문 중복 생성을 방지
- **재고·크레딧 방어**: GAS의 LockService, 주문 상태 검사, 재고·크레딧 최종 검증
- **주방 운영**: 주문 탭, 제공 완료 목록, 통계, 게스트/일반 재고 현황, 운영 설정
- **관리자 관리**: 이용자 크레딧, 간식 재고·가격·표시, 간식 순서, 관리자 로그
- **이용 신청**: `guest-apply.html`에서 사전 등록을 받고, 관리자 화면에서 승인·반려·중지·연락 완료·익명화 처리
- **PWA**: 서비스 워커와 화면별 매니페스트를 사용하며, 정적 자산을 캐시합니다. GAS API 요청은 캐시하지 않습니다.
- **접근성 피드백**: TTS, 버튼 효과음, 햅틱, 큰 터치 영역과 모바일 반응형 레이아웃

## 현재 아키텍처

```text
정적 HTML/CSS/JS
        │
        │ fetch / JSONP-like GAS web app 요청
        ▼
Google Apps Script
        ├── Google Sheets: 이용자·간식·주문·후기·운영설정·신청
        └── Google Drive: 이미지 업로드·썸네일 URL
```

### 프론트엔드

- `css/style.css`: 공통 색상, 버튼, 카드, 관리자 탭과 모바일 레이아웃
- `js/config.js`: GAS 웹앱 URL과 Mock 모드 설정
- `js/app.js`: 공통 API 호출, 인증 상태, TTS·햅틱·이미지 URL 처리
- 각 HTML 화면의 페이지 전용 JavaScript: 화면 상태와 렌더링 담당
- `service-worker.js`: 정적 파일 캐시와 오프라인 안내

### GAS 기능별 파일

Apps Script에서는 파일 순서가 실행 모듈을 나누는 것이 아니라, 유지보수 시 영역을 찾기 위한 번호 체계입니다.

| 파일 | 책임 |
| --- | --- |
| `00_Config.gs` | 시트명, 상수, 관리자 인증과 공통 설정 |
| `00_Setup.gs` | GAS 편집기에서만 실행하는 일회성 설정 함수 |
| `01_Router.gs` | 웹 요청 라우팅, 공개/관리자 API 분기 |
| `10_KakaoGuests.gs` | 카카오 인증과 게스트 프로필 |
| `11_AdminLog.gs` | 관리자 변경 로그 |
| `12_GuestApplications.gs` | 배달왔삼 이용 신청, 정원, 승인·익명화 |
| `20_Users.gs` | 일반 이용자와 크레딧 |
| `21_Snacks.gs` | 간식·가격·재고·표시 상태 |
| `30_GuestCredits.gs` | 게스트 크레딧 지갑 |
| `31_OrderShared.gs` | 주문 공통 헤더·변환·검증 보조 |
| `40_Orders.gs` | 주문 생성·조회·취소·제공·보관 |
| `50_Media.gs` | 이미지 업로드와 URL 처리 |
| `60_Settings.gs` | 운영 설정과 배달팀 정보 |
| `70_Reviews.gs` | 후기 등록·조회·답글·공개 상태 |
| `90_Diagnostics.gs` | 시트·헤더·스크립트 속성 진단 |

카카오 키와 관리자 토큰 같은 비밀값은 저장소에 넣지 않습니다. `00_Setup.gs`의 일회성 설정 함수도 새 GAS 프로젝트에서만 사용하고, 실행 후 비밀값이 포함된 코드를 GitHub에 복사하지 않습니다.

## 개인정보와 공개 주소 원칙

이 저장소는 공개 URL에서 동작하는 앱을 전제로 합니다.

- 일반 키오스크는 법적 실명 대신 `닉네임`을 기본 표시명으로 사용합니다.
- 일반 이용자 목록 API와 로컬 저장소는 현재 구조상 공개 키오스크 흐름의 일부이므로, 실명·개인 식별 사진을 넣는 것을 전제로 하지 않습니다.
- 주문·주방·전광판·빌지에는 배달왔삼 이용 신청의 연락처·상세 배달지를 복사하지 않습니다.
- `guest-apply.html`만 이름·관계·연락처·배달 장소를 사전 신청 목적으로 수집할 수 있습니다. 신청 정보는 `이용신청` 시트와 인증된 관리자 상세 화면에서만 사용합니다.
- 신청 개인정보는 브라우저 저장소에 보관하지 않습니다. 반려·중지 후 보관 기간이 지나면 관리자가 점검과 확인 문구를 거쳐 수동 익명화합니다.
- 현재 구조에서 일반 키오스크에 실명과 개인 사진을 적용하려면 공개 API, 이미지 접근, 로컬 저장, 자동 로그아웃을 별도로 보호하는 추가 설계가 필요합니다. 따라서 현재는 닉네임 기반 운영을 유지합니다.

## Google Sheets 구조

운영 DB의 실제 열 순서와 호환성은 [DB 구조 문서](docs/handoff/database-schema.md)를 기준으로 확인합니다.

| 시트 | 용도 |
| --- | --- |
| `이용자목록` | 일반 키오스크 이용자, 닉네임, 크레딧, 사용 여부, 이미지 URL |
| `간식목록` | 간식 ID, 이름, 가격, 이미지, 판매 여부, 재고, 표시 순서와 대상 |
| `주문내역` | 현재 주문, 주문번호, 수량, 차감, 취소·제공 상태, 게스트 식별값, `idempotencyKey` |
| `주문보관` | 운영자가 보관한 과거 주문. 현재 주문의 전체 열과 항상 동일하다고 가정하지 않음 |
| `관리자로그` | 관리자 변경 전후 값과 메모 |
| `운영설정` | 영업 상태, 크레딧·배달비, 배달팀 등 |
| `후기내역` | 후기, 공개 여부, 사진, 답글 |
| `게스트프로필` | 게스트 표시명과 마지막 배송지 |
| `게스트크레딧` | 날짜별 게스트 지갑과 사용 크레딧 |
| `이용신청` | 배달왔삼 사전 이용 신청과 승인 상태. `setupGuestApplicationSheet()`로 준비 |

이용 신청 계약은 19열이며, 기본 시범 정원은 5명입니다. 관리자가 신청 운영 설정에서 1~100명 범위로 조절할 수 있고, `PENDING + APPROVED` 신청이 정원을 차지합니다. 신청자 상태 조회 페이지나 자동 문자·카카오 발송은 현재 범위에 포함하지 않습니다.

## 문서와 작업 방식

작업을 시작할 때는 먼저 [handoff.md](handoff.md)를 읽습니다. 이 파일이 현재 작업의 유일한 입구이며, 세부 문서는 다음과 같이 나뉩니다.

- `docs/handoff/architecture.md`: 현재 화면·API·GAS 구조
- `docs/handoff/decisions.md`: 확정한 설계 원칙과 보류 결정
- `docs/handoff/database-schema.md`: 실제 DB 헤더와 코드 계약
- `docs/handoff/verification.md`: 배포 후 수동 검증 절차
- `docs/handoff/work-log.md`: 완료 작업만 기록하는 현재 로그
- `docs/handoff/*-archive-*.md`, `docs/handoff/handoff-archive.md`: 오래된 작업 로그·검토 문서
- `docs/handoff/asset-sources.md`: 이미지·아이콘·에셋 출처와 사용처
- `docs/admin-manual.md`: 관리자·주방·후기·이용 신청 운영 설명서

문서 점검은 다음 명령으로 실행합니다.

```powershell
node scripts/check-handoff.js
```

## 로컬 개발과 배포

1. 정적 파일은 Live Server 또는 다음 명령으로 실행합니다.

   ```powershell
   python -m http.server 8081
   ```

2. GAS 프로젝트에는 `gas/`의 파일을 같은 이름으로 반영합니다. 새 GAS 프로젝트에서는 필요한 일회성 `00_Setup.gs` 함수를 실행한 뒤 비밀값을 저장소에 남기지 않습니다.
3. GAS 웹앱을 배포하고 URL을 `js/config.js`의 `API_URL`에 설정합니다.
4. 카카오 로그인을 사용할 경우 GAS 스크립트 속성과 카카오 Redirect URI를 확인합니다.
5. 프론트엔드 변경 후 `service-worker.js`의 캐시 버전을 갱신하고, 강제 새로고침·PWA 재실행으로 새 정적 파일을 확인합니다.
6. 배포 전에는 PC·모바일 수동 검증, 기존 주문·재고·크레딧 회귀 검증, 신청 개인정보 노출 여부를 확인합니다.

### 주의

- GAS 변경은 배포 버전과 실제 운영 시트를 함께 확인해야 합니다.
- `getUsers` 등 공개 키오스크용 API에 실명·민감한 개인정보를 추가하지 않습니다.
- `ADMIN_TOKEN`, Kakao 키, Salt 등 비밀값은 GitHub에 커밋하지 않습니다.
- 주문·재고·크레딧 관련 변경은 프론트 버튼 잠금만 믿지 말고 GAS의 서버 검증과 멱등성 방어를 함께 확인합니다.
