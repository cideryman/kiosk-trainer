## Priority Queue

### P1 - 관리자 잠금 해제 UI 및 관리자 키 로그 보호 (구현 완료, 배포 검증 대기)

- **목적**: 작업 중 갑자기 나타나는 시스템 `prompt()`를 없애고, 관리자·주방·후기 화면에서 관리자 권한 상태와 입력 위치를 일관되게 제공합니다.
- **선행 보안 작업**: `doPost()`가 현재 POST 본문 전체를 `Logger.log()`에 기록하므로 `adminToken`이 실행 로그에 남지 않도록 제거하거나 마스킹합니다.
- **권장 UI**: 각 관리 화면의 색상 헤더 바로 아래에 얇은 공통 잠금 해제 바를 둡니다. 잠금 상태에서는 비밀번호 입력과 `잠금 해제`, 성공 후에는 상태 배지와 `잠그기`만 표시합니다.
- **검증 방식**: 입력한 키를 서버에서 먼저 확인하고 성공한 값만 `sessionStorage`의 `kioskAdminToken`에 저장합니다. 권한 거절 시 즉시 삭제합니다.
- **저장 원칙**: 앱이 관리자 키를 `localStorage`에 장기 저장하지 않습니다. 표준 비밀번호 입력 폼을 사용해 브라우저 저장 여부는 사용자가 선택하게 합니다.
- **기기 정책**: 직원 전용이고 화면 잠금이 적용된 관리 기기에서는 브라우저 비밀번호 저장을 허용할 수 있습니다. 이용자가 만지는 공용 키오스크에서는 저장하지 않습니다.
- **구조 개선**: 세 HTML에 중복된 `getAdminToken()`, 상태 표시, 권한 오류 초기화 코드를 공통 헬퍼로 묶을지 구현 전 검토합니다.
- **구현 결과**:
  - `js/app.js`에 공통 `AdminAuth` 헬퍼를 추가하고 세 화면의 상단 폼과 연결했습니다.
  - `verifyAdminAccess` 서버 검증 후에만 `sessionStorage`에 키를 저장하고, 잠그기·권한 거절 시 즉시 삭제합니다.
  - `gas/01_Router.gs`는 요청 본문 대신 action만 로그에 기록해 `adminToken`이 GAS 실행 로그에 남지 않도록 했습니다.
  - `service-worker.js` 캐시 버전을 `kiosk-cache-v131`로 올렸습니다.
  - 로컬 별도 포트에서 관리자·주방·후기 화면의 입력 폼 렌더링과 이벤트 연결을 확인했습니다.
- **남은 검증**: GAS 새 버전 배포 후 올바른 키·잘못된 키·잠그기·권한 거절·브라우저 비밀번호 저장 동작을 운영 기기에서 확인합니다.

### P2 - 비회원 게스트 주문표시명 하루 1회 정책

- **목적**: 같은 사람이 당일 주문마다 다른 표시명을 사용해 배달·주방·전광판에서 식별이 어려워지는 상황을 줄입니다.
- **현재 상태**: 로컬 게스트가 확정한 `localGuestDisplayName`은 같은 브라우저의 다음 주문에 재사용되지만 날짜 단위 확정 정책은 아직 없습니다.
- **권장안**: 첫 주문 완료 시 `{ dateKey, displayName }`을 저장하고 같은 날짜에는 자동 적용합니다. 다음 날에는 새 이름을 정할 수 있게 합니다.
- **설계 필요**: 오타 수정용 `표시명 변경` 경로, 당일 기존 주문 또는 진행 중 주문이 있을 때 변경 허용 범위, 안내 문구를 함께 결정합니다.
- **데이터 원칙**: 표시명은 로컬 기기에만 저장하고 배달지, 원본 카카오 식별자, 크레딧 정보는 추가 저장하지 않습니다.

### P3 - `placeOrder` 트랜잭션 정합성 검토

- **목적**: 주문 행 기록, 재고 차감, 일반·게스트 크레딧 차감 중간에 오류가 발생했을 때 부분 성공을 탐지하거나 복구할 필요가 있는지 판단합니다.
- **현재 판단**: 지금은 보류합니다. 멱등성은 동일 요청 재전송 방지이고 트랜잭션은 부분 실패 복구이므로 한 작업으로 섞지 않습니다.
- **착수 조건**: 주문 행은 있으나 재고나 크레딧 반영이 다르다는 실제 사례가 생기거나 운영 점검에서 불일치가 확인되면 P1으로 올립니다.
- **금지선**: 증거 없이 대규모 롤백 구조나 별도 원장 시트를 먼저 도입하지 않습니다.

### P4 - 후기 참여 정보 표시 검토

- 후기 상세 모달에 작성 횟수 등 참여 정보를 보여줄지 검토합니다.
- 카카오 연결 여부처럼 이용자 식별을 강화할 수 있는 정보는 노출 필요성과 개인정보 영향을 먼저 판단합니다.

### P5 - 미사용 CSS 및 잔여 코드 정리

- 화면 분리 과정에서 남은 미사용 선택자와 비활성 코드를 점진적으로 정리합니다.
- 운영 기능 변경과 섞지 않고, 화면별 회귀 검증이 가능한 작은 단위로 진행합니다.

- 클라이언트 `getUsers` 2분 캐시, 순차 `setTimeout` 폴링, 20초 API 타임아웃을 적용했습니다.

## Pending Manual Verification

- 배포 환경에서 관리자·주방·후기 화면의 `외부 화면 > 배달왔삼`이 실제 `guest.html`을 여는지 확인합니다.
- 예약 종료 또는 즉시 마감 시 `메뉴 미리보기`가 `menu.html?browse=guest` 읽기 전용 화면으로 열리고 수량·장바구니·주문 버튼이 없는지 확인합니다.
- 예약 종료 전에 시작한 주문은 종료 후 5분 안에 접수되고, 즉시 마감 후 제출과 예약 종료 5분 이후 제출은 거절되는지 확인합니다.
- 같은 `idempotencyKey` 강제 재전송 검증은 가능한 테스트 환경이 있을 때 수행하는 선택 항목입니다. 일반 버튼 잠금과 정상 주문 흐름은 검증 완료했습니다.
