## Priority Queue

### P13 - 배달왔삼 이용 신청·승인 관리 (구현 완료·배포 대기)

- **목적**: 복지관 봉사자·후원자와 관리자가 인정하는 관계자가 이용 가능 여부를 사전 신청하고, 관리자가 배달 장소를 확인한 뒤 수동으로 안내합니다. 신청은 메뉴 주문이 아니며 기존 주문 접근을 기술적으로 제한하지 않습니다.
- **구현**: `guest-apply.html`, 관리자 `신청` 탭, `gas/12_GuestApplications.gs`, `이용신청` A:S 19열 계약을 추가했습니다.
- **데이터 보호**: 이름·연락처·장소는 `이용신청`과 인증된 관리자 상세에서만 사용합니다. 브라우저에는 재시도용 `requestId`만 현재 탭에 임시 보관하고 성공 후 삭제합니다.
- **중복 방지**: `LockService` 안에서 `APP-YYYYMMDD-순번`을 생성하고 동일 `requestId`에는 기존 결과를 반환합니다. 익명화되지 않은 동일 연락처는 새 행을 만들지 않습니다.
- **상태**: `PENDING`, `APPROVED`, `REJECTED`, `INACTIVE`만 사용합니다. 반려·중지 후 30일이 지나면 관리자 확인을 거쳐 개인정보를 익명화합니다.
- **제외 범위**: 신청자 상태 조회, 대기열, 승인 코드, 자동 문자·카카오 발송은 도입하지 않습니다. 승인·반려 결과는 기관 담당자가 직접 연락합니다.
- **남은 작업**:
  1. 운영 DB를 백업합니다.
  2. GAS에 `00_Config.gs`, `01_Router.gs`, `12_GuestApplications.gs`, `90_Diagnostics.gs`를 반영합니다.
  3. `setupGuestApplicationSheet()`를 실행해 `이용신청` 19열과 누락 설정 보정 결과를 확인합니다.
  4. 신청 설정을 `마감`으로 둔 상태에서 관리자·공개 화면과 비인증 차단을 검증합니다.
  5. 개인정보 안내 문구를 기관 담당자가 확인한 뒤 신청을 엽니다.
- **자동 검증 완료**: GAS 결합 구문, HTML 인라인 스크립트, PC·390px·320px 레이아웃, 공개 Mock 접수, 관리자 목록·상세, 멱등 재전송, 동일 연락처 차단, 마스킹과 30일 만료일 생성을 확인했습니다.
- **운영 수동검증**: [배달왔삼 이용 신청 배포 검증](verification.md#배달왔삼-이용-신청-배포-검증-대기)을 따릅니다.

### P14 - 배달왔삼 1차 시범 신청 정원 관리 (구현 완료·배포 대기)

- **목적**: 초기 신청 규모를 작게 유지해 배달 장소·연락·주문 흐름을 안정적으로 검증합니다.
- **정원 설정**: `운영설정`의 `guestApplicationCapacity`를 사용하며 기본값은 5명입니다. 관리자가 1~100명 사이 정수로 조절할 수 있습니다.
- **정원 계산**: `PENDING + APPROVED`가 설정 정원을 차지하고 `REJECTED + INACTIVE`는 제외합니다.
- **정원 변경**: 현재 접수 인원보다 낮추면 기존 신청은 유지하고 신규 접수만 마감합니다. 다시 높이면 `이용 신청 받기`가 켜진 상태에서 자동으로 접수를 재개합니다.
- **최종 방어**: 화면의 남은 인원과 30초 설정 캐시는 안내용입니다. 신규 접수와 재승인은 `LockService` 안에서 원본 시트와 **현재 설정 정원**을 다시 확인합니다.
- **정원 초과**: `APPLICATION_FULL`을 반환하고 이름·연락처·장소를 저장하지 않습니다.
- **자동 검증 완료**: 기본 정원 5명 도달, 초과 미저장, 멱등 재전송, 반려 후 자리 재개방, 정원 축소·확대, 0·101·소수·문자 거절과 이전 프론트 호환성을 확인했습니다.
- **운영 수동검증**: 실제 `PENDING + APPROVED` 수와 관리자 모집 현황, 설정 정원 도달 시 공개 폼 마감, 정원 축소·확대와 입력 범위를 확인합니다.

### P16 - 관리자 인증 종료 버튼 의미 개선 (배포 완료·수동 검증 대기)

- 관리자·주방·후기 화면의 인증 종료 버튼을 `🚪 나가기`로 표시합니다.
- 버튼을 누르면 관리자 세션이 끝나고 전체 화면 로그인 게이트로 돌아갑니다.
- 툴팁과 접근성 이름은 `관리자 나가기`로 통일했습니다.
- **수동 검증**: 세 화면에서 버튼 문구·위치·툴팁, 클릭 후 게이트 복귀, 재로그인과 모바일 배치를 확인합니다.

### P17 - 모바일 간식 순서 변경 UI 개선 (배포 완료·수동 검증 대기)

- 모바일 순서 카드는 한 열로 표시하고 위·아래 이동 버튼을 주 조작으로 사용합니다. PC 드래그는 보조 조작으로 유지합니다.
- 첫 항목의 위로 이동과 마지막 항목의 아래로 이동은 비활성화합니다.
- 저장 실패 시 저장 전 순서로 다시 렌더링하며 일반·게스트 순서는 계속 분리합니다.
- **수동 검증**: PC 드래그, 390px·320px 위·아래 이동, 첫·마지막 경계, 긴 이름, 일반·게스트 분리와 저장 실패 복귀를 확인합니다.

### 수동 검증 - 배달왔삼 마감 경계

- 예약 종료 또는 즉시 마감 시 `메뉴 미리보기`가 `menu.html?browse=guest` 읽기 전용 화면으로 열리는지 확인합니다.
- 예약 종료 전에 시작한 주문은 종료 후 5분 안에 접수되고 그 이후에는 거절되는지 확인합니다.
- 즉시 마감 뒤 제출은 유예 없이 거절되는지 확인합니다.

### P7 - 후기 참여 정보 표시 검토 (검토 대기)

- 후기 상세 모달에 작성 횟수 등 참여 정보를 보여줄지 검토합니다.
- 카카오 연결 여부처럼 식별을 강화할 수 있는 정보는 노출 필요성과 개인정보 영향을 먼저 판단합니다.

## Deferred

### P15 - 배달왔삼 선택형 TTS 안내 (보류)

- 음성 안내는 선택 기능으로 제공하고 기본값은 끕니다.
- 시작·주문 완료·상태 변경·취소처럼 중요한 순간에만 제한적으로 안내합니다.
- 간식 클릭·폴링마다 반복 안내하거나 주문 상세정보를 상시 방송하지 않습니다.
- 실제 이용자에게 필요한 문구와 개인정보 노출 범위를 정한 뒤 착수합니다.

### P8 - 미사용 CSS 및 잔여 코드 정리 (보류)

- 참조되지 않는 CSS, 중복 인라인 스타일과 호출되지 않는 JavaScript를 작은 단위로 정리하는 유지보수 작업입니다.
- 시각적 회귀 위험이 있으므로 기능 변경과 섞지 않습니다.

### `placeOrder` 트랜잭션 정합성 구조 개편 (보류)

- 현재까지 주문 기록·재고·크레딧 간 실제 불일치는 관찰되지 않았습니다.
- 멱등성은 동일 요청 재전송 방지이고 트랜잭션은 부분 실패 복구이므로 한 작업으로 섞지 않습니다.
- 실제 불일치가 발생하거나 반복되는 실패 지점이 확인될 때 다시 검토합니다.
