## 변경 유형별 기본 원칙

- **문서만 변경**: Markdown 링크, 제목 구조, 최신 상태와 코드·DB 계약 일치 여부를 확인합니다. GAS 재배포나 서비스 워커 갱신은 필요하지 않습니다.

## 기능별 검증 절차

### P16 관리자 나가기 버튼

1. 관리자·주방·후기 화면에서 `🚪 나가기` 문구와 툴팁이 표시됩니다.
2. 버튼을 누르면 관리자 세션이 끝나고 전체 화면 로그인 게이트로 돌아갑니다.
3. 다시 로그인하면 해당 관리 화면이 정상적으로 초기화됩니다.
4. PC와 390px·320px에서 버튼 위치와 다른 기능 버튼 정렬이 유지됩니다.

### P17 모바일 간식 순서 변경

1. PC에서는 드래그, 모바일에서는 위·아래 버튼으로 순서를 바꿀 수 있습니다.
2. 첫 항목의 위로 이동과 마지막 항목의 아래로 이동은 비활성화됩니다.
3. 긴 간식명이 버튼을 밀거나 세로 글자로 갈라지지 않습니다.
4. 일반·게스트 순서가 서로 섞이지 않습니다.
5. 저장 실패 시 저장 전 순서로 복원됩니다.

### V1 배달왔삼 마감 경계

1. 예약 종료 또는 즉시 마감 상태에서 `메뉴 미리보기`가 `menu.html?browse=guest` 읽기 전용 화면으로 열립니다.
2. 읽기 전용 메뉴에는 수량 조절, 장바구니와 주문 확인 버튼이 없습니다.
3. 예약 종료 전에 시작한 주문은 종료 후 5분 안에 접수됩니다.
4. 예약 종료 5분 이후 제출은 거절됩니다.
5. 즉시 마감 이후 제출은 유예 없이 거절됩니다.

## 배달왔삼 이용 신청 배포 검증

### 배포와 데이터 계약

1. GAS 반영 전 운영 DB를 백업합니다.
2. `setupGuestApplicationSheet()` 실행 후 `이용신청` A:S 19열 헤더 순서를 확인합니다.
3. `운영설정`에 신청 설정 키와 `guestApplicationCapacity`가 생성되고 기존 관리자 저장값이 보존되는지 확인합니다.
4. 신청 설정이 `마감`일 때 공개 폼이 닫히고 관리자 안내 문구만 표시되는지 확인합니다.

### 공개 신청과 멱등성

1. 필수값 누락, 잘못된 연락처, 글자 수 초과, 동의 미체크와 허니팟 입력을 서버가 거절합니다.
2. 정상 신청에 `APP-YYYYMMDD-순번`이 생성되고 공개 성공 화면과 관리자 대기 목록에 같은 번호가 표시됩니다.
3. 같은 `requestId` 재전송은 행을 늘리지 않고 기존 신청번호를 반환합니다.
4. 다른 `requestId`의 동일 연락처 신청은 개인정보를 노출하지 않는 중복 안내만 반환합니다.
5. 설정 저장 후 30초 안에 공개 운영 안내와 희망 요일 선택지가 갱신됩니다.

### 관리자 처리와 개인정보

1. 비인증 상태에서는 관리자 목록·상세·상태 변경·설정·익명화 API가 모두 거절됩니다.
2. 목록에서는 연락처가 `010-****-1234` 형식이고 상세 모달에서만 전체 연락처·장소·동의 시각이 보입니다.
3. 승인·반려·중지·재승인·연락 완료·메모 저장이 반영됩니다.
4. 반려·중지 시 30일 만료일이 생성되고 재승인 시 만료일이 해제됩니다.
5. 만료 전에는 익명화되지 않고, 만료 후 `신청정보정리` 확인 문구로 익명화하면 개인정보·메모·`requestId`가 제거됩니다.
6. 연락처와 주소가 주문내역·주방·전광판·빌지·후기·GAS 실행 로그·관리자로그에 나타나지 않습니다.

### P14 신청 정원

1. 관리자 설정에서 1~100명 사이 정수만 저장되고 0·101·소수·문자는 거절됩니다.
2. 관리자 모집 현황의 접수 인원은 실제 `PENDING + APPROVED` 행 수와 일치합니다.
3. 설정 정원에 도달하면 공개 폼이 마감 안내로 바뀌고 초과 신청의 개인정보는 저장되지 않습니다.
4. 현재 접수 인원보다 정원을 낮춰도 기존 행은 유지되고 신규 신청과 정원 초과 재승인만 차단됩니다.
5. 정원을 다시 높이면 `이용 신청 받기`가 켜진 상태에서 공개 신청이 자동으로 재개됩니다.
6. 반려·중지로 자리가 생기면 다시 신청할 수 있습니다.
7. 정원이 찬 뒤에도 기존 `requestId` 재전송은 `APPLICATION_FULL`이 아니라 기존 신청 결과를 반환합니다.

### 화면과 기존 기능 회귀

1. PC·390px·320px에서 공개 소개·신청서와 관리자 목록·상세·설정에 가로 넘침, 세로 글자 분리와 터치 곤란이 없습니다.
2. 기존 일반 키오스크, 로컬·카카오 배달왔삼 주문, 크레딧, 취소와 후기 흐름이 이전과 동일합니다.

### P18 라벨 인쇄 검증

1. **빌지/라벨지 화면 동적 전환**: 
   - `print-bills.html`에서 인쇄 형식을 '일반 빌지'에서 '애니라벨 V3050'으로 바꿀 때, 화면이 라벨지 그리드(3열) 형식으로 자동 전환되고 `localStorage`에 모드가 기억되는지 확인합니다.
2. **개별 상품 낱개 분할**:
   - 동일 메뉴의 주문 수량이 2개 이상일 때(예: 아메리카노 2개, 쿠키 1개), 각각 낱개(아메리카노 2장, 쿠키 1장)로 라벨 카드가 분리 생성되는지 확인합니다.
   - 각 라벨 카드 내 메뉴 이름 우측에 `(순번/총수량)` 형식(예: `아메리카노 (1/3)`, `아메리카노 (2/3)`, `쿠키 (3/3)`)이 올바르게 매핑되는지 확인합니다.
3. **주문자 식별 및 정보 매핑**:
   - 낱개 분리된 각 라벨 카드마다 주문번호, 주문자명(닉네임), 수령방식, 배송지(배달 시) 정보가 빠짐없이 정상적으로 표시되는지 확인합니다.
4. **인쇄 정렬 오차 보정**:
   - `Ctrl + P`를 통해 인쇄 미리보기 화면을 열고, 2열과 3열 카드가 1열 카드 대비 우측으로 1mm씩 이동하여 출력 칼선 정렬 보정이 시각적으로 적용되는지 브라우저 개발자 도구 및 렌더링 결과로 확인합니다.

### p19-라벨-필터-및-오프셋-검증

1. **제공 완료 제외 토글 검증**:
   - `print-bills.html` 상단에 '제공 완료된 주문 제외' 체크박스가 표시되는지 확인합니다.
   - 체크박스를 활성화하면 이미 `제공완료 (Y)` 처리된 주문 항목들이 라벨지 및 체크리스트 인쇄 영역에서 즉시 제외되는지 확인합니다.
   - 체크박스를 비활성화하면 `제공완료` 상태인 주문들도 다시 나타나는지 확인합니다.
   - 제외 여부와 상관없이 품목 라벨의 순번 표시(예: `아메리카노 (1/3)`)는 원래 전체 주문 수량을 기준으로 일정하게 계산 및 유지되는지 검증합니다.
2. **시작 위치(오프셋) 지정 검증**:
   - 인쇄 형식이 '애니라벨 V3050'일 때만 시작 행(1~7) 및 시작 열(1~3)을 설정하는 UI가 나타나는지 확인합니다.
   - 2행 3열부터 인쇄하도록 지정한 경우, 첫 페이지의 앞의 5칸(1행 1~3열, 2행 1~2열)이 비어있고 테두리 없이 투명하게 출력되는지 확인합니다.
   - 전체 라벨 수가 A4 1페이지(21칸)를 넘어가면, 두 번째 페이지부터는 오프셋 없이 1행 1열부터 꽉 차서 출력되는지 확인합니다.
### P20 포장 체크리스트 인쇄

1. **배달/포장 체크박스 필터 동작**:
   - `print-bills.html` 상단에 '배달 주문 포함' 및 '포장 주문 포함' 체크박스가 각각 표시되는지 확인합니다.
   - 각 체크박스의 체크 상태에 따라 하단의 배달 체크리스트(`delivery-checklist-container`)와 포장 체크리스트(`takeout-checklist-container`)가 실시간으로 보이고 숨겨지는지 확인합니다.
   - 새로고침 시 이 설정값이 `localStorage`를 통해 정상 복원되는지 확인합니다.
2. **포장 체크리스트 데이터 정렬**:
   - 포장 주문 체크리스트가 배송지와 무관하게 **주문번호 및 주문 시간(timestamp)순**으로 시간 흐름에 맞춰 정렬되는지 확인합니다.
   - 닉네임, 주문번호, 메뉴, 수량이 올바르게 출력되는지 확인합니다.
3. **인쇄 시 페이지 분할(Page Break)**:
   - 브라우저 인쇄(Ctrl + P) 미리보기 화면에서 배달 체크리스트와 포장 체크리스트가 각각 별도의 A4 페이지 상단에서 시작하도록 물리적으로 분할 출력되는지 확인합니다.
4. **라벨지 모드 회귀 검증**:
   - 인쇄 형식을 '애니라벨 V3050'으로 설정했을 때 두 체크리스트 영역이 모두 정상적으로 화면 및 인쇄에서 숨겨지는지(`display: none`) 확인합니다.

## 선택 심화 검증

같은 사용자와 동일한 `idempotencyKey`로 같은 `placeOrder` 요청을 강제로 두 번 보낼 수 있을 때 주문 행, 재고와 크레딧이 한 번만 변경되고 두 번째 응답이 기존 `orderNo`와 `orderToken`을 반환하는지 확인합니다.
